# Security Policy 🛡️

Protecting users and contributors is a core priority for ThreatFusion.

This document explains how to report vulnerabilities responsibly and what to expect from our triage and response process.

## ✅ Supported Versions

Security fixes are prioritized for:

- the latest code in the default branch
- actively maintained release lines (if any are announced)

If a vulnerability affects older code, we may request an upgrade path to a supported version before issuing a fix.

## 🚨 How To Report a Vulnerability

Please do **not** report vulnerabilities through public GitHub issues.

Preferred channel:

- GitHub private security advisory for this repository

If private advisory flow is unavailable, contact maintainers privately with the details below.

Include as much of the following as possible:

- clear vulnerability description
- affected component(s) and file path(s)
- impact/severity assessment
- prerequisites and attack scenario
- step-by-step reproduction instructions
- proof-of-concept payload or script (if safe to share)
- suggested mitigation or patch idea (optional)

## 📬 What Happens After You Report

Our process generally follows these steps:

1. Acknowledge receipt.
2. Triage and validate the report.
3. Assess severity and scope.
4. Design and implement remediation.
5. Verify fix and prepare release/update.
6. Coordinate disclosure timing with reporter when applicable.

## ⏱️ Response Goals

We aim to:

- acknowledge valid reports promptly
- keep reporters informed on triage status
- communicate expected timelines where possible

Exact timelines may vary depending on complexity and maintainer availability.

## 🔒 Responsible Disclosure Expectations

We ask researchers to:

- avoid public disclosure before a fix is available or agreed disclosure date
- avoid data destruction, privacy violations, or service disruption
- test only what is necessary to demonstrate impact

## 🤝 Safe Harbor

Good-faith security research is welcome.

If you act in good faith and follow responsible disclosure practices, we will treat your work as authorized for the purpose of improving project security.

## 🙏 Recognition

We appreciate responsible reporters and may acknowledge contributions in release notes or project documentation, subject to reporter preference.
